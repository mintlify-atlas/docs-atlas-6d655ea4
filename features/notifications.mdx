title: "Notifications"
description: "Send transactional HTML emails from other microservices using the Ubik notification service's internal email API."

The notification service is a lightweight standalone microservice that delivers transactional emails over SMTP. It runs on port `8084` and is designed to be called by other microservices — it is not exposed directly to end users through the API gateway.

## Architecture overview

Other microservices communicate with the notification service using a `NotificationClient` component that posts to the internal `/notifications/email` endpoint. Communication is fire-and-forget: errors are logged and swallowed so that a failed email never blocks the calling service's main flow.

```
Payment Service ──► POST /notifications/email
Motel Management ──► POST /notifications/email
User Management ──► POST /notifications/email
```

## The email endpoint

### POST /notifications/email

Sends an HTML email to the specified recipient. The endpoint accepts a `NotificationRequest` body and returns `HTTP 200` on success.

<Note>
The endpoint checks for an `X-Internal-Request: true` header. While the check is currently commented out in the production code, callers should include this header so that the guard can be re-enabled without requiring changes to clients.
</Note>

### NotificationRequest fields

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `to` | `String` | Required, valid email format | Recipient email address |
| `subject` | `String` | Required, not blank | Email subject line |
| `message` | `String` | Required, not blank | Email body; rendered as HTML |

```bash
curl -X POST https://notification-service:8084/notifications/email \
  -H "Content-Type: application/json" \
  -H "X-Internal-Request: true" \
  -d '{
    "to": "guest@example.com",
    "subject": "Your reservation is confirmed — UBIK",
    "message": "<h2>Reservation confirmed</h2><p>Your stay at <strong>Hotel Casa Colonial</strong> from April 10 to April 13 is confirmed. Confirmation code: <strong>UBIK-2026-8842</strong>.</p>"
  }'
```

## SMTP configuration

The notification service uses Spring Mail with Gmail's SMTP server. The following environment variables must be set before the service starts:

| Variable | Description |
|----------|-------------|
| `MAIL_USERNAME` | Gmail address used as the sender (e.g. `ubikApp3@gmail.com`) |
| `MAIL_PASSWORD` | Gmail App Password (not your account password) |

The full SMTP configuration applied at runtime:

```yaml
spring:
  mail:
    host: smtp.gmail.com
    port: 587
    username: ${MAIL_USERNAME}
    password: ${MAIL_PASSWORD}
    properties:
      mail:
        smtp:
          auth: true
          starttls:
            enable: true
          connection-timeout: 5000
          timeout: 5000
          write-timeout: 5000
```

<Tip>
Generate a Gmail App Password at **Google Account → Security → 2-Step Verification → App passwords**. Using a plain Gmail password will not work because Google blocks less-secure app access by default.
</Tip>

<Warning>
The `spring.mail.health.enabled` property is set to `false`. This prevents Spring Actuator's health check from opening a live SMTP connection on startup, which would cause the service to fail its health probe if Gmail is temporarily unavailable.
</Warning>

## HTML email rendering

The `NotificationService` uses `MimeMessageHelper` with `setText(body, true)` to render the `message` field as HTML. All emails sent from Ubik are styled inline HTML — plain-text fallback is not currently provided.

Example of the payment confirmed email body generated by the payment service:

```html
<div style="font-family: Arial, sans-serif; padding: 30px; background:#f4f6f9;">
  <div style="max-width:600px; margin:0 auto; background:#fff; border-radius:8px; padding:30px;">
    <h2 style="color:#27ae60; text-align:center;">Payment confirmed</h2>
    <p>Your payment was processed successfully.</p>
    <p><strong>Reservation:</strong> #88</p>
    <p><strong>Total paid:</strong> $450,000.00 COP</p>
    <p style="color:#777; font-size:12px; text-align:center;">Thank you for choosing UBIK</p>
  </div>
</div>
```

## Calling the notification service from another microservice

Use the `NotificationClient` pattern already present in the payment and motel management services. The client posts to the notification service base URL, which is configured via:

- Payment service: `services.notification.url`
- Motel management: `notification.service.url` (defaults to `http://notificationService:8084`)

```java
// Asynchronous, non-blocking — errors are swallowed
notificationClient.sendEmailAsync(
    guestEmail,
    "Your reservation is confirmed — UBIK",
    htmlBody
).subscribe();
```

All calls use `subscribeOn(Schedulers.boundedElastic())` so that the blocking SMTP send does not tie up the reactor thread pool.
